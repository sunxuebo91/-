# 客户订单中心功能规划

> 版本：v1.0 | 日期：2026-03-29

---

## 一、背景与目标

客户（用户）通过微信小程序授权手机号登录后，系统可以将其手机号与 CRM 后端合同数据进行匹配。  
匹配成功后，客户可在小程序「我的」页面查看自己的服务合同、下载合同、在线签约、以及确认阿姨上户。

---

## 二、整体流程图

```
客户打开小程序
  → 授权手机号登录（loginByPhone）
  → 手机号存入云数据库 users 集合
  → 「我的」页面出现「我的订单」入口
  → 订单列表（按手机号从 CRM 拉取合同）
  → 订单详情（展示合同所有信息）
      ├── 合同下载
      ├── 去签约（打开签约链接）
      └── 确认上户（通知后端改状态）
```

**安全链路说明：**  
客户端 → 云函数（用 openid 验证身份，查出手机号）→ CRM 后端（用共享密钥 + 手机号查合同）  
客户端永远不直接调用 CRM 接口，避免身份伪造。

---

## 三、【后端 CRM】需要做什么

### 3.1 合同/订单表新增字段

| 字段名 | 类型 | 说明 | 为什么加 |
|--------|------|------|---------|
| `signingUrl` | String | 电子签约链接（如法大大） | 客户需要点链接去完成签约 |
| `contractFileUrl` | String | 合同 PDF 下载地址 | 客户需要下载保存合同 |
| `onboardStatus` | Enum | `pending` / `confirmed`，默认 `pending` | 记录客户是否已确认阿姨上户 |
| `onboardConfirmedAt` | DateTime | 客户点击确认上户的时间 | 留存记录，CRM 可追溯 |
| `onboardConfirmedBy` | String | 确认人手机号 | 验证是本人操作 |

**确认合同表已有字段（如无需补充）：**

| 字段名 | 说明 |
|--------|------|
| `customerName` | 客户姓名 |
| `customerPhone` | 客户手机号（身份匹配核心） |
| `serviceType` | 服务类型枚举（zhujia-yuer 等） |
| `serviceFee` | 服务费/中介费 |
| `nannySalary` | 阿姨工资 |
| `nannyName` | 阿姨姓名 |
| `nannyPhone` | 阿姨电话 |
| `startDate` | 服务开始日期 |
| `status` | 合同状态（pending_sign/signed/active/ended） |

### 3.2 新增接口（共 3 个）

**接口鉴权方式：X-Service-Secret（共享密钥）**  
> 为什么不用 JWT Token？因为小程序用户没有 CRM 账号体系的 Token，且接口只允许云函数调用，不对客户端开放，所以用服务端密钥做机器间鉴权更简单安全。

---

#### 接口 1：获取客户合同列表

```
GET /api/miniprogram/contracts?phone={手机号}
Header: X-Service-Secret: <共享密钥>
```

响应示例：
```json
{
  "success": true,
  "data": [
    {
      "id": "合同ID",
      "customerName": "张女士",
      "serviceType": "zhujia-yuer",
      "serviceTypeText": "住家育儿嫂",
      "serviceFee": 15000,
      "nannySalary": 8000,
      "nannyName": "王阿姨",
      "nannyPhone": "138****0000",
      "startDate": "2026-04-01",
      "status": "active",
      "onboardStatus": "pending",
      "signingUrl": "https://...",
      "contractFileUrl": "https://..."
    }
  ]
}
```

**为什么这样设计：**  
按手机号查合同是最自然的身份匹配方式。合同本来就绑定了 `customerPhone`，用它做索引即可，无需引入 openid 这个微信专属概念到 CRM 系统。

---

#### 接口 2：获取单个合同详情

```
GET /api/miniprogram/contracts/:id?phone={手机号}
Header: X-Service-Secret: <共享密钥>
```

**后端必须做的权限校验：**  
`contracts.customerPhone === 传入的 phone`，不一致返回 403。

**为什么要做这个校验：**  
防止客户 A 通过猜合同 ID 看到客户 B 的合同信息（越权访问）。

---

#### 接口 3：确认上户

```
POST /api/miniprogram/contracts/:id/confirm-onboard
Header: X-Service-Secret: <共享密钥>
Body: { "phone": "13800138000" }
```

**后端处理逻辑：**
1. 验证密钥
2. 查合同，校验 `customerPhone === phone`（防越权）
3. 校验 `onboardStatus === 'pending'`（防重复确认，已确认直接返回成功）
4. 更新：`onboardStatus = confirmed`，`onboardConfirmedAt = NOW()`，`onboardConfirmedBy = phone`

**为什么客户端来触发而不是员工：**  
确认上户是客户视角的动作（"我确认阿姨已经来我家了"），作为客户的主动确认行为留存在合同里，比员工单方面填写更可信，也减少员工漏操作。

## 四、【小程序前端】需要做什么

### 4.1 新建云函数 `contractService`

**为什么要加一层云函数而不直接调 CRM：**  
云函数运行在微信服务器上，可以从微信上下文拿到不可伪造的 `openid`，用它查出用户手机号，再带共享密钥调用 CRM。客户端如果直接调 CRM，手机号可以被伪造，没有安全保障。

支持的 action：

| action | 说明 |
|--------|------|
| `getMyContracts` | 查当前用户的合同列表 |
| `getContractDetail` | 查单个合同详情 |
| `confirmOnboard` | 确认上户 |

云函数内部逻辑：
```
① 从微信上下文取 openid（不可伪造）
② 查云数据库 users，取出该用户的 phone
③ phone 为空 → 返回错误"请先绑定手机号"
④ 携带 phone + X-Service-Secret 调用 CRM 对应接口
⑤ 返回数据给小程序
```

---

### 4.2 修改 `pages/profile/index`（我的页面）

新增「我的订单」菜单项，显示条件：
- 用户已登录（有手机号）
- `role === 'customer'`（员工不显示，员工有独立的简历管理入口）

**为什么员工不显示：**  
员工也是有手机号的用户，如果不加角色判断，员工也会看到这个入口，但员工不是合同里的客户方，毫无意义。

---

### 4.3 新建 `pages/myOrders/index`（订单列表页）

展示内容：
- 合同状态标签（待签约 / 已签约 / 服务中 / 已结束）
- 服务类型 + 阿姨姓名
- 服务开始日期
- 上户状态标签（待上户 / 已上户）

---

### 4.4 新建 `pages/myOrders/detail`（订单详情页）

展示内容：

```
客户姓名 / 手机号
服务类型（中文展示）
服务费（中介费）
阿姨工资
阿姨姓名 / 阿姨电话（可点击拨打）
服务开始日期
合同状态
上户状态
```

操作按钮（按状态动态显示）：

| 按钮 | 显示条件 | 行为 |
|------|---------|------|
| 下载合同 | `contractFileUrl` 不为空 | `wx.downloadFile` + `wx.openDocument` |
| 去签约 | `signingUrl` 不为空 且 `status === pending_sign` | 跳转 WebView 打开签约链接 |
| 确认上户 | `onboardStatus === pending` 且 `status === active` | 弹窗二次确认后调云函数 |

**确认上户交互细节：**
```
点击「确认上户」
  → wx.showModal 弹窗："确认阿姨已正式开始上户服务？"
  → 用户点确认 → 调用 contractService.confirmOnboard
  → 成功：按钮变为「已确认上户」并置灰，显示确认时间
  → 失败：toast 提示错误原因
```

## 六、给后端开发的核心说明摘要

> 转发给后端同学时，重点说清楚以下几点：

1. **鉴权方式**：接口不用用户 JWT Token，用固定的服务端共享密钥（`X-Service-Secret`），只有我们的云函数会调这几个接口，不对外暴露
2. **身份匹配**：用 `customerPhone` 做客户身份的唯一标识，不引入微信 openid
3. **必须做越权校验**：每个接口都要验证传入的 phone 和合同里的 customerPhone 一致，否则返回 403
4. **确认上户幂等**：已是 `confirmed` 状态再次请求，直接返回成功，不报错
5. **字段补全**：确认合同表已有 customerName / nannyName / nannyPhone / serviceFee / nannySalary，没有的话这次一起补上
